Defect Investigation

Real-world defect analysis with root cause identification, impact assessment, and resolution validation.

BUG-2025-014 High
Loan Eligibility Income Miscalculation
  • ProblemCustomer loan eligibility incorrectly failed for applicants with multiple income sources.
  • InvestigationQueried the database via SQL to verify stored income values against the API payload in Postman.
  • Root CauseThe backend API schema for the eligibility endpoint dropped the secondary income array during payload processing.
  • ValidationExecuted API regression testing with varied income combinations to confirm total calculation.
Summary
Secondary income sources were dropped during the eligibility API request, causing valid loan applications to be rejected.
Environment
Staging API v2.4, Windows 11, Chrome.
Steps to Reproduce
1. Submit loan application via API with primary_income and secondary_income. 2. Check eligibility status endpoint.
Expected Result
Total income should aggregate both fields and approve eligibility.
Actual Result
Only primary_income was calculated; eligibility failed.
Investigation Process & Evidence

Isolated the issue using Postman and SQL verification. Below is the logs extraction and DevTools API response verification showing the omitted object block:

HTTP/1.1 200 OK | GET /eligibility/assess?id=APP_90212
// EXPECTED RESPONSES CONTAINED secondary_income ARRAY
{ "applicant_id": "APP_90212", "primary_income": 85000.00, "secondary_income": [ {"source": "rent", "amount": 12000.00} ] <-- MISSING IN ACTIVE PAYLOAD }
🎫 JIRA: BUG-2025-014 ● ACTIVE (TRIAGED)
Summary: Eligibility endpoint dropping secondary income list during calculation mapping.
Priority: High | Component: core-eligibility-api
Reproduction steps: 1. Run eligibility test suite with applicant having rental income. 2. Verify response array size is empty.
Root Cause Analysis
API contract validation showed that the secondary_income object was not included in the eligibility service response, despite being successfully stored in the database.
Resolution Validation
The development team updated the API schema definition. I re-ran the API test suite and verified the database calculation logs to confirm correct aggregation.
Regression Scope
Full API regression suite for the Loan Approval Workflow.
Lessons Learned
Added schema validation assertions in Postman to catch contract changes earlier.
Tools Used
Postman, SQL, Jira, Chrome DevTools.

VAL-2025-041 High Impact
Monthly Commission Validation & Reporting Automation
  • ProblemManual validation of monthly commission deductions for all insurance agents required 7–8 working days.
  • InvestigationExtracted commission data using SQL through SAP BusinessObjects and validated against 10-12 business attributes.
  • Root CauseDiscrepancies were identified where calculated commission values did not match predefined commission rules for specific business attribute combinations.
  • ValidationVerified corrected calculations, generated automated exception reports, and presented the final validation report to senior management.
Business Problem
The manual validation of monthly commission deductions across all insurance agents required 7–8 working days, causing delays in monthly executive reporting.
Environment
SAP BusinessObjects, Life Asia, Python, SQL, Advanced Excel.
Validation Workflow
1. Execute SQL queries to extract data. 2. Validate combinations of Agent Type, Policy Type, Premium Mode, Policy Term, and Commission Slab.
Expected Result
Actual extracted commission values must perfectly match the expected values defined in the predefined commission rule master file.
Actual Result
Manual validation was highly time-consuming and increased the risk of missing complex business rule discrepancies.
Investigation Process & Evidence

Analyzed discrepancies by writing database reconciliation scripts. Below is the verified exception row showing the rule mismatch:

exception_report_snapshot.xlsx (Mismatched Slab Rows)
Policy No Product Code Expected % Actual % Status
POL90812 TERM_LIFE_02 12.50% 10.00% DISCREPANCY
SQL Audit query executed:
SELECT agent_id, SUM(comm_amt) FROM actual_comm GROUP BY agent_id HAVING SUM(comm_amt) <> expected_total;
Discrepancy Analysis
Rather than internal software bugs, analysis revealed discrepancies where calculated commission values did not match predefined commission rules for highly specific combinations of business attributes.
Resolution Validation
Developed a Python automation solution that read extracted data and rule master files, applied all validation rules automatically, compared Expected vs Actual commissions, and generated exception reports.
Business Impact
Reduced validation effort from approximately 7–8 working days to just 1 day. Enabled faster monthly reporting and the timely presentation of validated reports to the Vice President.
Lessons Learned
Quality Assurance extends beyond UI testing; automating complex business rule validation is critical for financial data integrity.
Tools Used
Python, SQL, SAP BusinessObjects, Life Asia.

BUG-2025-028 Medium
Partial Refund Workflow Failure
  • ProblemProcessing a partial refund for orders with discount codes resulted in an incorrect refund amount.
  • InvestigationReproduced the issue via the UI and tracked the network payload using Browser DevTools to verify logic.
  • Root CauseNetwork payload analysis confirmed the front-end client was incorrectly calculating the tax multiplier before the discount subtraction.
  • ValidationDesigned and executed positive and negative UI scenarios using Selenium to verify boundary values.
Summary
Customers returning single items from a multi-item discounted order were refunded slightly more than their actual purchase price.
Environment
QA Environment, Order Management Dashboard, Selenium WebDriver.
Steps to Reproduce
1. Create an order with 2 items and a 10% discount code. 2. Process a partial refund for Item A.
Expected Result
Refund amount should equal (Item A price - 10%) + tax on that specific amount.
Actual Result
Refund amount calculated as (Item A price + tax) - 10%.
Investigation Process & Evidence

Monitored the refund execution flow using Browser DevTools. Captured the following client request console logs and trace calculations showing floating point precision rounding errors:

[console.log] client_payment_controller.js:L102
item_base_price = 899.10
discount_code_applied = 0.10 // (10%)
calculated_refund_subtotal = 809.1900000000001 // Floating point error
tax_rate = 0.05 // (5%)
final_refund_payload_sent = 849.65 (Expected: 849.64)
Selenium assertion failure track:
java.lang.AssertionError: Expected refund amount [849.64] but was [849.65]
Root Cause Analysis
The request payload generated by the client contained incorrect refund values, indicating an issue in the client-side calculation. It was applying the tax multiplier before the discount subtraction, breaking the standard accounting order of operations.
Resolution Validation
After the calculation logic was corrected, I executed an automated Selenium suite covering full refunds, partial refunds, and boundary values (e.g., $0.01 items).
Regression Scope
All payment and refund workflows within the OMS dashboard.
Lessons Learned
Added specific boundary and fractional penny test cases to the automation framework.
Tools Used
Selenium WebDriver, Browser DevTools, Jira.

philosophy

Principles I Follow

The beliefs that shape how I approach quality engineering.

01
Test Early
Defects cost 10× more to fix after release. I push quality left.
02
Verify Business Logic
Technical correctness means nothing if the business rule is wrong.
03
Validate Data
What the UI shows and what the database holds must always agree.
04
User Impact First
Severity is defined by real-world consequence, not technical complexity.
05
Automate Wisely
Automate the repetitive. Reserve human judgment for exploratory testing.
06
Prevent Regression
Every bug fixed is a new test case. Regressions are preventable failures.
07
Document Clearly
A defect report is only as useful as it is reproducible. Clarity is quality.
